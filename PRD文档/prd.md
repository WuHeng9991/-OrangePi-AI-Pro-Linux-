# 1. 产品概述

## 1.1 项目背景

当前项目面向大三学生的实习简历场景，目标不是做企业级播放器，而是基于开源原型完成一个“可运行、可演示、可讲解”的中等体量项目。项目应突出 C 语言开发能力、多媒体框架理解、Linux 应用开发与板端部署调试经验。

## 1.2 产品目标

·  完成本地音视频文件的打开、播放、暂停、停止、进度显示与拖动跳转。

·  当前阶段优先在 OrangePi AI Pro 的桌面环境中完成播放器图形界面开发、基础播放链路验证与远程可视化联调。

·  后续阶段再完成 HDMI 本地显示验证，并继续推进 ALSA / 音频输出路径验证。

·  在代码结构上完成从单文件原型到模块化项目的重构，便于简历描述与面试讲解。

·  控制开发范围，使项目在 2 周左右完成 MVP，并具备可扩展空间。

## 1.3 目标用户


| **主要用户：**项目开发者本人，用于学习与求职展示。          | **使用环境：**当前主要开发验证环境为 Ubuntu/openEuler + OrangePi AI Pro + 可远程访问的桌面环境；后续本地显示验证环境为 OrangePi AI Pro + HDMI 显示器。 |
| ----------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **展示对象：**校招/实习面试官、课程答辩老师、技术社团同学。 | **典型输入：**本地 MP3、WAV、FLAC、MP4 等多媒体文件。            |

## 1.4 非目标（本阶段不做）

·  不做在线音乐平台接入、账号系统、云端媒体库同步。

·  不做复杂皮肤系统、插件市场、歌词联网抓取等重功能。

·  不以“支持所有编码格式”为目标，以常见本地文件可播放为验收基线。

| R2       | 播放控制         | 支持播放、暂停、停止三类基本控制，按钮状态与播放状态保持一致。               | P0         | 点击后行为正确，无明显状态错乱。         |

## 3.2 关键用户流程

### 当前开发 / 验证流程

1. 用户先远程连接到 OrangePi 开发板桌面环境。
2. 用户启动程序后，界面初始化并显示播放器主窗口。
3. 用户点击“打开文件”，选择本地媒体文件。
4. 程序创建并切换 GStreamer pipeline，开始播放媒体。
5. 播放过程中周期性刷新位置、时长和进度条。
6. 用户可执行暂停、恢复、停止或拖动进度条跳转。

### 后续扩展验证流程

1. 接入 HDMI 外接显示器并切换到本地显示环境。
2. 验证播放器窗口、视频画面与基础交互是否正常。
3. 如需板端适配，再继续验证 HDMI / 耳机口 / ALSA 等音频输出路径。

# 4. 界面与交互设计

## 4.1 主界面布局建议


| **区域**   | **组件**            | **说明**                                             |
| ---------- | ------------------- | ---------------------------------------------------- |
| 顶部栏     | 文件打开 / 设备选择 | 放置“打开文件”“选择音频输出设备”等控件。         |
| 中部       | 视频显示区          | 视频文件时显示画面；音频文件时显示占位图或应用图标。 |
| 右侧/底部  | 播放列表            | 显示目录扫描结果，支持点击切换。                     |
| 底部控制区 | 播放/暂停/停止      | 保持按钮直观、图标或文本均可。                       |
| 底部控制区 | 进度条 + 时间标签   | 显示当前位置 / 总时长，并支持拖动跳转。              |


| **交互原则** 按钮数目控制在基础可用范围，避免学生项目过度堆叠功能。先保证播放链路、状态同步和板端显示稳定，再考虑界面美化与附加能力。 |
| ------------------------------------------------------------------------------------------------------------------------------------- |

## 4.2 交互细节要求

·  停止播放后，进度条应回到起点，时间标签复位为 00:00 / 总时长。

·  拖动进度条时，应避免刷新线程与 seek 行为冲突，必要时进入“用户正在拖动”状态。

·  打开新文件后，应释放旧 pipeline 相关资源，避免音视频串流残留。

·  播放结束后应显示已结束状态，并允许用户重新播放。

# 5. 技术方案与系统设计

## 5.1 技术选型


| **层级**   | **方案**  | **说明**                                          |
| ---------- | --------- | ------------------------------------------------- |
| 语言       | C         | 满足目标公司“偏 C 语言”岗位方向，突出基础能力。 |
| GUI        | GTK3      | 轻量、Linux 原生感强，适合学生项目图形界面。      |
| 多媒体框架 | GStreamer | 便于搭建播放 pipeline，并支持后续 sink/设备扩展。 |
| 音频接口   | ALSA      | 用于板端音频输出验证与设备切换。                  |
| 构建系统   | CMake     | 满足跨平台/多文件工程构建需求。                   |

## 5.2 模块拆分


| **模块**      | **文件建议**   | **职责**                                  |
| ------------- | -------------- | ----------------------------------------- |
| 应用入口      | main.c / app.c | 初始化 GTK 与 GStreamer，组织程序主流程。 |
| 播放控制      | player.c       | 封装播放、暂停、停止、seek 等控制接口。   |
| Pipeline 管理 | pipeline.c     | 创建元素、动态 pad 链接、处理 bus 消息。  |
| 界面层        | ui.c           | 创建窗口、按钮、进度条、列表与相关回调。  |
| 播放列表      | playlist.c     | 目录扫描、媒体过滤、上一首/下一首逻辑。   |

## 5.3 建议 Pipeline（示意）


| **推荐播放链路** source/uridecodebin → 动态 pad 分流 → 音频分支（queue → audioconvert →  audioresample → autoaudiosink/alsasink）；视频分支（queue → videoconvert →  gtksink/gtkglsink）。 |

# 8. 风险与应对


| **编号** | **风险点**                    | **影响**                           | **应对措施**                                   |
| -------- | ----------------------------- | ---------------------------------- | ---------------------------------------------- |
| K1       | GStreamer 动态 pad 链接不稳定 | 导致部分媒体无法正常播放。         | 先保证常见格式；增加日志并用简单样例验证。     |
| K2       | seek 与进度刷新冲突           | 造成拖动跳转后界面抖动或回跳。     | 增加“用户拖动中”状态，临时暂停刷新。         |
| K3       | 开发板音频输出路径不一致      | HDMI 有图无声或设备切换失败。      | 使用 ALSA 设备枚举/测试命令逐项验证。          |
| K4       | 项目范围膨胀                  | 导致最终无法在投递前形成完整演示。 | 严格按 MVP → 板端适配 → 增强功能的顺序推进。 |
| K5       | 源码重构后引入回归问题        | 原本能播的文件播放异常。           | 每次重构后做回归测试并保留稳定分支。           |

# 9. 验收标准

### 当前阶段验收

·  程序可在 OrangePi 板端桌面环境中稳定编译运行，并可通过远程连接方式看到桌面与播放器界面。

·  支持打开并播放本地媒体文件；播放、暂停、停止功能正确。

·  进度条可随播放更新，并支持拖动 seek。

·  代码完成基础模块化拆分，README 中包含编译、运行与测试说明。

·  可录制一段 1–2 分钟的演示视频用于简历投递或面试展示。

### 后续阶段验收

·  接入 HDMI 外接显示器后，可正常完成本地显示验证。

·  可完成至少一种本地音频输出方式验证（优先 HDMI，其次耳机口）。
