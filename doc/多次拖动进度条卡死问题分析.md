# 多次拖动进度条后音视频卡死问题分析

## 1. 问题现象

在播放器正常播放过程中，如果连续多次拖动进度条跳转到不同位置，容易出现以下现象：

- 界面状态仍显示为“播放中”
- 画面停止刷新
- 声音停止输出
- 再次点击播放按钮通常不能恢复
- 需要先点击 `Stop`，再点击 `Play`，播放器才会恢复正常

这个问题最容易在频繁 seek 的情况下触发，单次拖动通常不明显。

## 2. 初始实现方式

问题出现时，底层播放链路采用的是手工拼接的 GStreamer 管线，核心结构如下：

```text
uridecodebin
  -> video_queue -> videoconvert -> videosink
  -> audio_queue -> audioconvert -> audioresample -> audiosink
```

这套实现本身可以完成基础播放、暂停、停止和 seek，但在重复 seek 时稳定性不足。

## 3. 现象排查过程

### 3.1 第一阶段判断

一开始怀疑是上层 UI 状态机问题，例如：

- `playing / is_seeking / seek_done / seek_resume_playback` 状态错乱
- seek 过程中状态判断过严，导致后续拖动被误拦截
- seek 后重复调用 `set_state()`，把 pipeline 推进了不一致状态

因此先做了这些修正：

- 清理进度条拖动逻辑
- 去掉 seek 后多余的状态切换
- 补充 `ASYNC_DONE`、`STATE_CHANGED`、`CLOCK_LOST`、`LATENCY` 等日志
- 增加播放进度看门狗，检测“状态正常但位置不前进”的情况

### 3.2 日志结论

后续日志证明，问题并不只是 UI 状态机层面：

- seek 请求已经成功发出
- bus 已收到 `ASYNC_DONE`
- pipeline 已收到 `STATE_CHANGED ... new=PLAYING`
- `LATENCY` 也已经重新计算成功
- 但 `query_position()` 停在某个时间点不再前进

也就是说：

- GStreamer 状态机“表面上”已经恢复到 `PLAYING`
- 但底层真实的数据流、解码链或时钟推进已经卡住

这就是为什么界面上看起来已经回到播放状态，但实际上画面和声音都不再继续。

## 4. 根因分析

根因可以总结为一句话：

**手工拼接的低层播放管线在重复 seek 后进入了“状态已恢复、但流未继续推进”的卡死状态。**

更具体地说：

- `GST_STATE_PLAYING` 只能说明 pipeline 状态切换完成
- 它不能保证底层音视频流一定还在持续推进
- 在多次 seek、flush、重新同步时，手工拼接的解码链更容易出现分支停滞
- 一旦某次 seek 后音视频分支或时钟没有真正恢复，位置查询会停住，表现为“假播放”

而 `Stop -> Play` 之所以能恢复，是因为：

- `Stop` 会把 pipeline 拉回 `READY`
- `Play` 会重新启动整条播放链
- 相当于重新建立了一次解码、同步和输出流程

这不是从根上解决问题，而是通过重启播放链把卡死状态冲掉。

## 5. 为什么前面的轻量修复不够

前面做过几轮补丁，它们是必要的，但不充分：

- 修正 seek 状态机：解决了明显的状态字段不一致问题
- 增加 `CLOCK_LOST` 处理：补齐了时钟丢失后的恢复路径
- 增加 `LATENCY` 处理：补齐了延迟重新计算
- 增加看门狗：确认了“PLAYING 但 position 不动”的真实故障形态

这些工作帮助我们把问题定位清楚，但没有彻底消除底层播放链在重复 seek 后卡死的问题。

原因在于：

- 问题已经不在“有没有发出 seek”
- 也不在“状态有没有切到 PLAYING”
- 而是在手工播放链本身对重复 seek 的鲁棒性不足

## 6. 最终解决方案

最终方案是：

**将底层播放管线从手工拼接实现切换为 GStreamer 的 `playbin`。**

## 6.1 改造思路

保留现有工程结构和 UI，不改上层交互逻辑，只替换底层播放核心：

- 原来：自己管理 `uridecodebin + queue + convert + sink`
- 现在：使用 `playbin` 统一管理解封装、解码、时钟、seek 和同步

同时保留现有项目需要的可配置能力：

- 保留自定义音频输出 sink
- 保留 GTK 视频显示控件
- 保留 bus 消息处理
- 保留状态标签、进度条、调试日志、看门狗

## 6.2 为什么 `playbin` 能解决

`playbin` 的优势不在于“功能更多”，而在于“播放主链更成熟”：

- 它本身就是 GStreamer 提供的高层播放组件
- 内部已经处理了更完整的播放、seek、同步和恢复路径
- 对常见媒体播放场景的兼容性和稳定性更好
- 可以减少我们自己维护低层管线细节带来的状态耦合和边界问题

这次问题的本质是“重复 seek 时低层链路卡死”，因此继续在手工管线上打补丁，收益已经不高；直接切到更稳定的高层播放组件，是更务实的解决方案。

## 7. 本次代码调整要点

本次修复涉及的核心点如下：

- 底层播放核心改为 `playbin`
- `uri` 直接设置到 `playbin`
- `audio-sink` 继续使用项目当前配置逻辑创建
- `video-sink` 继续复用当前 GTK 视频显示方案
- 保留并继续使用 bus 消息处理逻辑
- 保留故障诊断日志和播放进度看门狗，便于后续继续观察稳定性

## 8. 最终结果

改造后重新测试：

- 连续多次拖动进度条
- 多次跳转到不同位置
- 音视频能够继续正常播放
- 不再出现必须 `Stop -> Play` 才能恢复的情况

说明这次修复已经命中主要根因。

## 9. 经验总结

这次问题的关键经验有三点：

### 9.1 不能只看状态，要看进度是否真的前进

播放器显示 `PLAYING`，不代表底层播放链一定正常。  
真正可靠的判断是：

- `query_position()` 是否持续前进
- 音视频输出是否持续刷新

### 9.2 重复 seek 是播放器稳定性的高风险场景

播放器在正常顺序播放时可能没有问题，但一旦进入频繁拖动、频繁 flush、频繁同步恢复场景，低层实现的脆弱点会很快暴露。

### 9.3 对“标准播放器场景”优先使用高层组件

如果需求是“本地媒体播放 + 暂停 + 停止 + seek + GTK 显示 + 可配置音频输出”，优先使用 `playbin` 这类成熟高层组件，通常比手工拼接底层播放链更稳，也更节省维护成本。

## 10. 后续建议

- 保留当前日志与看门狗一段时间，继续观察 seek 稳定性
- 后续若需要进一步减少调试噪声，可在功能稳定后把日志降级或加开关控制
- 如果未来要接入板端硬解、专用 sink 或更复杂的滤镜链，再评估是否需要从 `playbin` 回退到定制 pipeline
- 在没有明确必要前，不建议为了“可控性”重新回到手工拼接播放主链
