# 二次开发计划文档

## Context

**本次二次开发以开源原型 **`open-pipe-media-player` 为底座，目标是按 PRD 将项目演进为一个适合 OrangePi AI Pro 板端演示与简历展示的 Linux 本地媒体播放器。当前阶段优先基于已可远程访问的开发板桌面环境进行开发与验证，并且已经可以看到桌面，因此现阶段的重点是先完成图形界面联调、基础播放链路验证与 MVP 功能开发；HDMI 外接显示屏接入后的本地显示验证，以及 HDMI/ALSA 音频输出验证，统一放在后续阶段推进。**

**结合现状：**

* **README 说明项目当前是基于 C / GTK3 / GStreamer / CMake / POSIX thread 的开发中原型。**
* `draft4.c` 已具备 GTK 窗口、视频显示、Play/Pause/Stop、`uridecodebin` 动态解码、音视频双分支、bus 消息处理等基础能力。
* `CMakeLists.txt` 已提供 `HWACC` 编译开关，可作为 HDMI/图形输出路径扩展基础。
* **当前缺口主要是：没有文件选择器、没有进度条与 seek、没有时间标签、没有 ALSA 输出切换、没有播放列表、错误提示较原始、代码结构集中在单文件。**

**推荐策略：以 **`draft4.c` 为唯一主线增量演进，优先复用已有播放链路与动态 pad 处理逻辑，避免推倒重写；按“远程桌面开发验证 → MVP 功能开发 → HDMI 本地显示验证 → 音频输出验证 → 增强功能”的顺序推进，保证每阶段都有可运行成果。

## 推荐实施方案

### 阶段 1：跑通原型并建立二开基线

**目标：确认当前原型可作为后续开发底座，并确认开发板桌面环境已可远程接入且可见，能够作为当前 GUI 开发与验证的基础。**

**任务：**

1. **以 **`draft4.c` 为主程序基线，确认 `README.md` 中依赖与构建流程可在目标环境复现。
2. **记录 PC Ubuntu 与 OrangePi AI Pro 环境差异，特别是 GStreamer 插件、GTK sink、图形桌面会话与音频输出链路差异。**
3. **确认当前开发方式基于远程连接到开发板桌面环境进行，明确 SSH 会话与桌面会话的关系。**
4. **明确主线代码与历史草稿边界：**`drafts/draft1-2.c`、`drafts/draft3.c` 仅保留为参考，不继续叠加功能。

**复用点：**

* `draft4.c` 中现有的 `CustomData` 上下文
* `pad_added_handler` 动态 pad 分流逻辑
* `handle_message` 的 bus 消息处理骨架
* **当前音频链路 **`queue -> audioconvert -> audioresample -> autoaudiosink`
* **当前视频链路 **`queue -> videoconvert -> gtksink/gtkglsink`

### 阶段 2：完成 V1 MVP 功能

**目标：交付 PRD 中的 P0 能力，形成可演示的基础播放器。当前阶段的功能开发、界面联调和基础验证优先在远程桌面环境下完成，不等待 HDMI 外接显示器接入。**

**任务：**

1. **实现本地文件打开**
   * **在 UI 中增加“打开文件”入口。**
   * **用统一的 URI 处理逻辑替代当前命令行拼接 **`file://` 的粗糙方式。
   * **若未选择文件，程序保持空闲态，不自动拉起默认网络视频。**
2. **完善播放控制**
   * **保持播放、暂停、停止按钮与真实播放状态一致。**
   * **停止后重置播放位置和状态。**
   * **播放结束后允许重新播放。**
3. **实现进度显示与拖动跳转**
   * **增加进度条与时间标签。**
   * **周期性刷新当前位置 / 总时长。**
   * **支持 seek，并处理“用户拖动中”状态，避免刷新与 seek 冲突。**
4. **增强状态提示**
   * **将 GStreamer 错误、EOS、打开失败转化为界面可见提示。**
   * **对不支持格式、文件不存在等情况给出明确反馈。**

**建议实现顺序：**

1. **文件打开**
2. **时间标签**
3. **进度条刷新**
4. **seek 跳转**
5. **状态提示与边界行为修正**

### 阶段 3：完成 OrangePi 板端适配亮点

**目标：先完成 HDMI 外接显示器接入后的本地显示验证，再继续推进音频输出路径验证，形成更完整的板端适配亮点。**

**任务：**

1. **完成 HDMI 本地显示验证**
   * **在 OrangePi AI Pro 上接入 HDMI 外接显示器，确认本地图形输出可用。**
   * **验证播放器窗口、视频画面与基础交互可在本地显示设备上正常呈现。**
2. **增加音频输出策略**
   * **在默认输出基础上，为 ALSA 指定设备输出预留配置项或参数入口。**
   * **逐步从 **`autoaudiosink` 演进到支持 `alsasink` 的可切换方案。
3. **完成 HDMI / 耳机口音频场景验证**
   * **在完成本地显示验证后，再验证至少一种音频输出方式（优先 HDMI，其次耳机口）的可用性。**
4. **保留回退机制**
   * **若指定设备不可用，则回退到默认输出路径，并打印明确日志。**
5. **记录板端调试命令**
   * **将 ALSA 设备查看、切换、验证命令整理进 README，作为演示材料一部分。**

**复用点：**

* `CMakeLists.txt` 中 `HWACC` 开关思路
* `draft4.c` 中视频 sink 初始化路径

### 阶段 4：进行模块化重构

**目标：从单文件原型过渡到可维护、可讲解的项目结构。**

**按 PRD 推荐拆分为以下模块：**

* `main.c` / `app.c`：程序入口、初始化、主流程组织
* `player.c`：播放、暂停、停止、seek 等控制接口
* `pipeline.c`：元素创建、动态 pad 链接、bus 消息处理
* `ui.c`：窗口、按钮、进度条、标签、文件选择器等控件与回调
* `playlist.c`：目录扫描、媒体过滤、上一首/下一首（若进入 V1.2）

**重构原则：**

1. **先按职责抽函数，再拆文件，避免一次性大重构。**
2. **每次拆分后都要保证当前播放链路仍可运行。**
3. **保留共享上下文结构，逐步从 **`CustomData` 演进为更清晰的应用状态对象。

### 阶段 5：按时间推进增强功能

**目标：在 MVP 和板端适配稳定后，补齐项目观感与展示价值。**

**可选增强项：**

1. **目录扫描与播放列表（PRD V1.2）**
2. **上一首 / 下一首**
3. **媒体信息显示（至少文件名、总时长）**
4. **更清晰的异常提示与状态文案**
5. **README、运行说明、演示录屏脚本整理**

**若时间受限，严格按 PRD 范围控制原则执行：优先交付 V1 MVP，再推进 HDMI 本地显示验证与音频输出验证，最后再考虑播放列表等增强项。**

## 需要重点复用的现有实现

* `draft4.c` 中 `play_cb` / `pause_cb` / `stop_cb`：可作为控制接口雏形
* `draft4.c` 中 `create_ui`：可扩展为完整 UI 层
* `draft4.c` 中 `pad_added_handler`：继续用于音视频动态 pad 链接
* `draft4.c` 中 `handle_message`：继续作为状态同步与错误处理中心
* `draft4.c` 中现有 pipeline 元素创建与链接流程：作为后续重构基础，而不是重写播放链路

## 主要风险与处理

1. **seek 与进度刷新冲突**
   * **增加“用户拖动中”状态，拖动期间暂停自动刷新，拖动完成后执行 seek 并恢复刷新。**
2. **打开新文件导致旧 pipeline 残留**
   * **切换媒体前先停止并释放旧 pipeline 相关资源，再创建/重置新链路。**
3. **板端 ALSA 输出路径差异**
   * **先在命令行工具层面确认设备，再在播放器内逐步接入设备切换。**
4. **模块化重构带来回归问题**
   * **每次重构后都回归本地文件播放、暂停/停止、时间刷新、退出清理。**

## 验证方案

### 当前阶段验证

1. **远程桌面环境验证：**
   * **确认已可远程连接到 OrangePi AI Pro 的桌面环境，并且可以看到桌面。**
   * **确认板端桌面会话可访问，SSH 会话与图形桌面会话关系清晰。**
2. **图形链路验证：**
   * **在板端桌面会话下验证 GTK / GStreamer 图形链路可运行。**
   * **保留 `DISPLAY=:0` 与 `XAUTHORITY` 等关键环境信息，便于后续复用。**
3. **功能验证：**
   * **MP3 / WAV / FLAC / MP4 至少各验证一种常见样本。**
   * **播放、暂停、停止、停止后重播、播放结束后重播。**
   * **进度条随播放推进更新。**
   * **拖动到中间和末尾附近均可正常跳转并继续播放。**
   * **文件不存在、格式不支持、打开失败时界面有明确提示且程序不崩溃。**

### 后续阶段验证

1. **HDMI 本地显示验证：**
   * **接入 HDMI 外接显示器后，确认本地图形输出可用。**
   * **确认播放器窗口、视频画面与基础交互可在本地显示设备上正常呈现。**
2. **音频输出路径验证：**
   * **使用 ALSA 相关命令确认设备后，验证至少一种音频输出路径。**
   * **保留关键日志，记录设备切换结果与失败原因。**

### 工程验证

1. **默认构建可通过。**
2. `HWACC=OFF` 构建可通过并能正常播放。
3. `HWACC=ON` 构建至少能完成元素创建/回退逻辑验证。
4. **模块拆分后 README 中包含编译、运行、测试说明。**
