# issue.md

## 1. 项目概述

这个项目是一个运行在 Linux / OrangePi AI Pro 开发板上的本地媒体播放器，项目名为 `open-pipe-media-player`。

它的目标不是做一个功能花哨的大型播放器，而是做一个：

- 能在 OrangePi 板端正常运行
- 能播放本地音视频文件
- 带有图形界面
- 支持打开文件、播放、暂停、停止、拖动进度条
- 便于演示、复盘和继续二次开发

从当前代码和阶段文档来看，这个项目已经从早期的单文件原型，演进成了一个结构清晰的多文件工程。

---

## 2. 项目整体架构

这个项目可以理解成 4 层：

### 第 1 层：程序入口层
负责启动 GTK 和 GStreamer，并把整个程序跑起来。

- `src/main.c`

### 第 2 层：应用编排层
负责管理播放器当前状态，把“界面操作”和“底层播放”串起来。

- `src/app.c`
- `include/app.h`

### 第 3 层：播放控制与播放管线层
负责真正控制播放、暂停、停止、拖动，以及音视频输出方案。

- `src/player.c`
- `include/player.h`
- `src/pipeline.c`
- `include/pipeline.h`

### 第 4 层：界面层
负责窗口、按钮、进度条、标签、文件选择框这些用户能直接看到和操作的部分。

- `src/ui.c`
- `include/ui.h`

简单理解就是：

- `main.c`：把程序开起来
- `ui.c`：把界面画出来
- `app.c`：负责“协调”
- `player.c`：负责“播放控制命令”
- `pipeline.c`：负责“底层音视频播放方案”

## 3. 每个源文件是干什么用的

下面按“小白能快速看懂”的方式解释。

### 3.1 `src/main.c`
这是程序总入口。

它做了 3 件事：

1. 初始化 GTK（图形界面库）
2. 初始化 GStreamer（音视频播放库）
3. 调用应用初始化、构建播放链路、运行主循环

可以理解成：

> 程序总开关，负责把整台机器发动起来。

---

### 3.2 `include/app.h`
这里定义了整个程序最核心的状态结构 `CustomData`。

这个结构里保存了：

- GStreamer 的播放对象
- 当前是否正在播放
- 是否已加载媒体
- 是否正在 seek（拖动跳转）
- 当前播放位置、总时长
- 音频输出配置
- GTK 的窗口、按钮、进度条、标签等控件

可以理解成：

> 这是整个程序的“总控制台”和“共享状态仓库”。

谁要知道“现在播到哪了”“窗口控件在哪”“当前音频输出是什么”，基本都从这里拿。

---

### 3.3 `src/app.c`
这是整个应用的“中控层”。

它主要负责：

- 初始化程序状态
- 读取音频输出相关环境变量
- 创建播放管线
- 打开媒体文件
- 定时轮询 GStreamer bus 消息
- 更新界面时间和进度条
- 打印调试日志
- 做播放卡死监测（watchdog）

它不是直接负责“画界面”，也不是直接负责“解码播放”，而是负责把两边串起来。

可以理解成：

> UI 说“用户点了播放”，底层说“我收到状态变化了”，中间都要经过 `app.c` 来协调。

---

### 3.4 `src/player.c`
这是播放器控制层，负责最基础、最直接的播放命令：

- `player_play`：播放
- `player_pause`：暂停
- `player_stop`：停止
- `player_seek`：跳转到指定时间
- `player_query_position`：查询当前位置
- `player_query_duration`：查询总时长

可以理解成：

> 这是一个“遥控器命令层”。

它不关心按钮长什么样，也不关心窗口怎么画，它只关心：

- 播放
- 暂停
- 停止
- 跳转

---

### 3.5 `src/pipeline.c`
这是项目里最关键的底层播放实现层。

它负责：

- 创建 GStreamer 播放核心
- 配置音频输出 sink
- 配置视频输出 sink
- 把 `playbin` 和音视频输出绑起来
- 处理 bus 消息（错误、播放结束、状态变化、异步完成、时钟丢失、延迟变化等）
- 做资源释放

可以理解成：

> 这是“真正让音视频播起来”的核心位置。

这个文件决定了项目到底使用什么播放方案、音频从哪里出、视频显示到哪里去。

---

### 3.6 `src/ui.c`
这是 GTK 图形界面层。

它负责：

- 创建窗口
- 创建 Open / Play / Pause / Stop 按钮
- 创建进度条
- 创建时间标签和状态标签
- 处理按钮点击事件
- 处理进度条按下和释放事件
- 打开文件选择框
- 刷新界面显示

可以理解成：

> 这是用户能直接看到和操作的“门面层”。

---

### 3.7 头文件的作用

- `include/app.h`：定义全局共享状态和应用层接口
- `include/player.h`：声明播放控制接口
- `include/pipeline.h`：声明底层播放管线接口
- `include/ui.h`：声明界面相关接口

头文件可以简单理解为：

> “说明书 + 对外接口表”

谁能调用什么函数、需要什么参数，都先在头文件里声明。

## 4. 当前项目使用了什么技术方案

## 4.1 图形界面方案

项目使用的是：

- **GTK3** 作为图形界面库

它负责：

- 创建窗口
- 创建按钮
- 创建进度条
- 创建标签
- 响应鼠标点击和拖动

GTK 在这个项目里的角色是：

> “把播放器界面显示出来，并处理用户操作。”

---

## 4.2 音视频播放方案

项目使用的是：

- **GStreamer 1.0** 作为音视频播放框架

GStreamer 是 Linux 下很常见的多媒体框架，它负责：

- 解封装
- 解码
- 音视频同步
- seek（拖动跳转）
- 输出到音频设备和视频窗口

---

## 4.3 当前底层播放核心方案

### 旧方案
早期使用的是手工拼接低层播放链，结构类似：

```text
uridecodebin
  -> video_queue -> videoconvert -> videosink
  -> audio_queue -> audioconvert -> audioresample -> audiosink
```

这套方案能工作，但重复 seek 时稳定性不够。

### 新方案
当前已经切换为：

- **`playbin` 作为底层播放核心**

`playbin` 是 GStreamer 提供的高层播放器组件，内部已经帮我们处理了很多复杂的事情：

- 自动解封装
- 自动解码
- 音视频同步
- 常见播放场景下更稳定的 seek 处理

这次切换的核心意义是：

> 不再自己手搓一条复杂且容易在边界场景卡死的播放链，而是改用 GStreamer 更成熟的高层播放方案。

---

## 4.4 视频显示方案

当前项目的视频显示分两种路径：

### 方案 A：`gtkglsink + glsinkbin`
这是偏向 GL 路径的视频显示方式。

特点：

- 更适合某些图形加速场景
- 在 `HWACC_ENABLED` 开启时优先尝试

### 方案 B：`gtksink`
这是回退方案，也是当前更稳妥的 GTK 视频显示方式。

特点：

- 更直观
- 更容易验证 GTK 视频窗口是否正常显示
- 当 GL 路径失败时回退使用

### 当前实际逻辑
在 `src/pipeline.c` 中：

- 如果启用了 `HWACC_ENABLED`，优先尝试 `gtkglsink + glsinkbin`
- 如果失败，就回退到 `gtksink`
- 如果没启用 `HWACC_ENABLED`，直接使用 `gtksink`

也就是说：

> 项目已经具备“优先尝试更强方案，失败就自动回退到稳妥方案”的能力。

---

## 4.5 音频输出方案

当前项目默认使用：

- `autoaudiosink`

它的意思是：

> 让 GStreamer 自动选择当前系统最合适的音频输出方式。

在当前环境里，`autoaudiosink` 实际上通常会走：

- `pulsesink`

此外，项目还支持通过环境变量切换：

- `OPMP_AUDIO_SINK`
- `OPMP_AUDIO_DEVICE`

例如：

- 指定使用 `alsasink`
- 指定 `device=default`

如果用户配置的 sink 创建失败，代码会自动回退到 `autoaudiosink`。

如果该 sink 不支持 `device` 属性，程序会打印清晰日志，而不是默默失败。

这套设计的价值是：

- 默认情况简单易用
- 调试时又能切换到指定音频方案
- 对板端设备验证友好

---

## 4.6 状态更新与消息处理方案

项目通过 **GStreamer Bus** 来监听播放过程中的事件。

当前已经处理的主要消息包括：

- `ERROR`：播放出错
- `EOS`：播放结束
- `DURATION_CHANGED`：总时长变化
- `ASYNC_DONE`：异步操作完成，常见于 seek 后
- `STATE_CHANGED`：播放状态变化
- `CLOCK_LOST`：时钟丢失
- `NEW_CLOCK`：收到新时钟
- `LATENCY`：延迟重新计算

这说明项目不是“只会播”，而是已经具备比较完整的状态反馈和调试基础。

---

## 4.7 进度条与 seek 方案

当前 seek 方案不是“滑一下就直接乱跳”，而是做了状态保护。

进度条相关逻辑包括：

- 按下进度条时先判断当前是否允许跳转
- 如果上一次 seek 还没完成，就拒绝新的 seek
- 只有当前媒体支持拖动时才允许 seek
- 拖动释放后，才真正发出 seek 请求
- seek 过程中会显示“跳转中”状态

这套逻辑的作用是：

- 避免用户乱拖导致状态打架
- 避免还没完成上一次跳转又来一次新跳转

---

## 4.8 播放卡死监测方案（watchdog）

项目在 `src/app.c` 里加入了播放进度看门狗。

它的核心思路是：

- 不只看 GStreamer 状态是不是 `PLAYING`
- 还看当前位置 `position` 是否真的在前进

如果长时间停在一个位置不动：

- 先记录日志
- 再尝试做一次轻量恢复：`PAUSED -> PLAYING`

这个设计很重要，因为它说明开发过程中已经意识到：

> “状态看起来正常”和“底层真的在继续播放”不是一回事。

## 7. 已解决的问题整理

下面按照你要求的格式统一整理。

---

## 问题 1：SSH 会话中无法直接打开图形界面

### 问题
在 SSH 终端中直接运行程序时，可能出现：

```bash
Gtk-WARNING **: cannot open display:
```

用户会误以为是程序有问题，实际上很多时候只是运行环境不对。

### 解决方法
原因是：

- SSH 会话和桌面图形会话是分开的
- 当前 shell 没有绑定正确的 `DISPLAY` 和 `XAUTHORITY`

处理方法：

```bash
env DISPLAY=:0 XAUTHORITY=/var/run/lightdm/root/:0 \
  ./build/open-pipe-media-player
```

或者直接在桌面终端里运行程序。

### 总结学到了什么
图形程序能不能启动，不只取决于代码是否正确，还取决于：

- 当前会话是不是图形桌面会话
- `DISPLAY` 和 `XAUTHORITY` 是否绑定正确

### 下次应该怎么做，怎么避免
- 在验证 GTK 程序前，先确认当前是不是桌面环境
- 如果是 SSH，会先确认 `DISPLAY`、`XAUTHORITY`
- 文档里提前写清楚两种运行方式，避免后续反复踩坑

---

## 问题 2：指定 ALSA 硬件设备时失败

### 问题
使用类似下面命令时，可能失败：

```bash
gst-launch-1.0 audiotestsrc num-buffers=1 ! alsasink device=plughw:CARD=LINKLIKE,DEV=0
```

常见现象是：

- 设备 busy
- 播放失败

### 解决方法
根因通常不是播放器代码写错，而是：

- PulseAudio 已经占用了底层 ALSA 硬件设备

排查命令：

```bash
fuser -v /dev/snd/*
pactl info
aplay -l
aplay -L
```

当前项目中的处理方式是：

- 默认保留 `autoaudiosink`
- 允许通过环境变量切到 `alsasink`
- 如果指定 sink 创建失败，就自动回退
- 如果 `device` 属性不支持，就打印日志提示

### 总结学到了什么
“能不能出声”不是一个单纯的播放器代码问题，它还涉及：

- 系统当前默认音频服务
- PulseAudio / ALSA 的关系
- 设备是否被占用
- 板端声卡枚举是否正确

### 下次应该怎么做，怎么避免
- 先验证默认链路 `autoaudiosink`
- 再验证 `alsasink device=default`
- 最后才尝试更底层的 `plughw`
- 遇到 busy 时先查占用，不要立刻怀疑播放器逻辑

---

## 问题 3：多次拖动进度条后音视频卡死

### 问题
播放器正常播放时，如果连续多次拖动进度条，会出现：

- 界面状态还是“播放中”
- 但画面不动
- 声音不响
- 再点播放通常恢复不了
- 往往要 `Stop` 再 `Play` 才能恢复

这说明出现了一种“看起来在播，实际上没播”的假播放状态。

### 解决方法
最开始怀疑是 UI 状态机问题，于是先补了这些东西：

- 清理 seek 状态逻辑
- 增加 `ASYNC_DONE`、`STATE_CHANGED`、`CLOCK_LOST`、`LATENCY` 日志
- 增加播放进度看门狗
- 增加轻量恢复路径

这些工作有帮助，但没有从根上解决问题。

最终确认的根因是：

> 旧的手工拼接低层 GStreamer 播放链，在重复 seek 后，可能进入“状态已经回到 PLAYING，但底层数据流没有继续推进”的卡死状态。

最终采用的解决方案是：

- **把底层播放核心从手工拼接管线切换为 `playbin`**

也就是：

- 保留现有 UI
- 保留状态标签、进度条、bus 消息处理
- 保留音频输出配置能力
- 但把真正的底层播放主链换成更成熟稳定的 `playbin`

修复后，连续多次拖动进度条，音视频可以继续正常播放。

### 这次进一步修复了什么
在后续板端实测里，又遇到了一类更隐蔽的情况：

- `seek` 完成后 bus 日志里已经回到 `PLAYING`
- 状态标签也显示“播放中”
- 但 `query_position()` 长时间停在同一个时间点
- 这时再点 `Pause`，日志里还可能短暂看到内部 `play=1`

这说明问题已经不再是“能不能切到 `PLAYING`”，而是：

> 管线状态恢复了，但底层播放位置没有继续推进，进入了 seek 后的“假播放”状态。

这次代码层面的修复主要做了 3 件事：

- 给每次 `seek` 记录最近请求时间，明确区分“普通播放卡住”和“seek 后立即卡住”
- watchdog 检测到“刚 seek 过且位置不前进”时，不再只做 `PAUSED -> PLAYING`
- 而是优先在当前位置补发一次 `FLUSH | ACCURATE` 的原位 `re-seek`，强制底层重新对齐播放点

如果原位 `re-seek` 仍然失败，才回退到原先的轻量恢复路径：

- `PAUSED -> PLAYING`

另外，这次还顺手修正了播放状态同步细节：

- `player_play()` 成功后及时同步 `playing = TRUE`
- `player_pause()` 成功后及时同步 `playing = FALSE`
- `ERROR / EOS / READY` 时清理最近一次 seek 的上下文，避免 watchdog 误判旧状态

这样处理之后，恢复链路就变成了：

1. 用户拖动进度条，正常发起 seek
2. 如果 seek 后位置继续前进，不做任何额外处理
3. 如果 seek 后出现“状态在播但位置不动”，watchdog 先做原位 `re-seek`
4. 只有原位恢复无效时，才退回 `PAUSED -> PLAYING`

实测结果是：

- 之前日志里那种 `STATE_CHANGED` 已回到 `PLAYING`，但位置卡在 `0:00:49` 不动的情况已经恢复正常
- 连续拖动进度条后，音视频可以继续往前播放
- `Pause` 时内部状态与界面状态的错位也明显减少

### 总结学到了什么
1. **不能只看状态，要看进度是不是真的在前进**
   `PLAYING` 不代表底层流一定健康，真正可靠的判断是：
   - 位置是不是持续变化
   - 音视频是不是持续输出

2. **重复 seek 是播放器稳定性的高风险场景**
   单次拖动没问题，不代表频繁拖动也没问题。

3. **标准播放器场景优先选成熟高层组件**
   如果需求是常规媒体播放，不一定非要手工拼复杂底层管线。高层组件很多时候更稳定、更省维护成本。

### 下次应该怎么做，怎么避免
- 设计播放器时，优先评估 `playbin` 这种成熟高层方案是否足够
- 对频繁 seek 场景单独做压力测试，而不是只测正常顺序播放
- 调试时不要只盯状态字段，要同时观察 `query_position()` 是否前进
- 对“状态恢复了但流不走”的情况，提前准备日志和看门狗
